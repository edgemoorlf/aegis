# PLAN.md

> A data-centric, capability-based substrate for multi-agent systems.
> Protect the data, not the sandbox.

## Project Name (working title)

**Aegis** — a data-plane security and consistency layer for multi-agent systems.

*(Alternatives to consider: `Capla`, `Ledger-Agent`, `Provenant`, `Warden`.)*

---

## Motivation

The prevailing approach to agent safety is **execution sandboxing** — isolate the
process or VM so misbehaving code cannot harm the host. This is the wrong primary
defense for three reasons:

1. **It protects the wrong layer.** The asset worth protecting is *data*, not the
   host OS. Sandboxes contain actions; they do not govern who may read, append, or
   exfiltrate data.
2. **The breach often happens upstream.** If context is sent to an external LLM API
   to generate code, the sensitive data has already left the trust boundary before
   any sandbox runs. Containing the output is theater.
3. **It ignores multi-agent consistency.** Per-agent isolation makes coherent shared
   state *harder*, not easier. Sandboxes say nothing about stale reads, memory
   divergence, or conflicting world-views across agents.

This project inverts the model: **the data layer is the security and consistency
boundary.** Compute is assumed untrusted and transient. Data is immutable,
versioned, redundant, and access is governed by capabilities — which, critically,
can now be *generated by an LLM and verified by a verifier agent*, removing the
historical reason capability-based security never scaled.

---

## Design Principles

1. **Never delete.** All writes are append-only and versioned. History is permanent
   and queryable (time-travel). Data survives any agent failure or misbehavior.
2. **Data-centric, not action-centric.** Protect data at the storage/access layer,
   independent of what code runs.
3. **Capability-based access.** Every data operation requires a capability token
   scoping data, operations, time window, and agent identity. No token, no access —
   regardless of executed code.
4. **LLM-generated, agent-verified capabilities.** Policy is expressed in natural
   language; an LLM emits the capability specification; a separate verifier agent
   audits it for over-permissioning and policy contradiction before issuance.
5. **Audit log as ground truth.** An independent, append-only, tamper-evident log is
   the authoritative record. Agent memories are derived views, never the source of
   truth. Cross-agent discrepancies are resolved against the log.
6. **Local-first inference.** The architecture assumes the LLM runs inside the trust
   boundary (on-prem / VPC / local). External APIs are an explicitly allowlisted,
   audited exception — not the default.
7. **Versioned memory with provenance.** Every agent memory records which agent
   wrote it, from which data version, at what time. Divergence is detectable.

## Architecture

                        │           Policy (NL)            │
                        └───────────────┬──────────────────┘
                                        │
                          ┌─────────────▼─────────────┐

                          └─────────────┬─────────────┘

        │                                                                     │

### Components

#### 1. Immutable Store (data plane)
- Append-only, versioned storage with time-travel queries.
- **Build on:** Apache Iceberg or Delta Lake for tabular data; an event-sourced log
  (e.g. on top of object storage / a log like Redpanda/Kafka) for agent events.
- Redundancy and robustness handled here (replication, snapshots), satisfying the
  "never delete" principle architecturally rather than by policy.

#### 2. Capability Broker (access plane)
- Issues, revokes, and expires capability tokens.
- Token schema (sketch): `{ data_selector, operations[], time_window, agent_id, issued_by, signature }`.
- **Build on:** OpenFGA or Cerbos for the enforcement engine; SPIFFE/SPIRE for
  workload (agent) identity; biscuit or macaroons for attenuable capability tokens.

#### 3. Capability Generator (LLM)
- Reads data schema + agent task description + org policy (natural language).
- Emits a candidate capability specification.
- Runs against a **local model** by default (llama.cpp / Ollama / vLLM).

#### 4. Verifier Agent
- Independent critic that audits the generated capability: over-permissioning,
  least-privilege violations, policy contradictions.
- Pattern: critic / constitutional loop. Must be a *separate* model context from the
  generator to avoid self-justification.

#### 5. Audit Log (ground-truth plane)
- Append-only, tamper-evident (hash-chained / Merkle), **write-only from the agents'
  perspective** — agents cannot read-modify-write it.
- Records every capability issuance, every data access, every agent action.
- Authoritative source for reconstructing ground truth and resolving discrepancies.
- **Build on:** a transparency-log design (e.g. Trillian-style Merkle log) or an
  immutable append-only table with periodic checkpoint signing.

#### 6. Memory Layer (cognitive plane)
- Agent memories stored with provenance: `(agent_id, source_data_version, timestamp)`.
- Cross-agent reads go through the Capability Broker like any other data access.
- Divergence detection: compare an agent's memory snapshot against the audit log.

---

## Multi-Agent Consistency Model

- Agents never share **mutable** state directly. They share **versioned, immutable
  snapshots** (MVCC applied to agent cognition).
- When two agents disagree, the system does **not** mediate between their memories.
  It queries the audit log and reconstructs authoritative ground truth.
- Write conflicts are surfaced explicitly (Git-merge-conflict style), never silently
  reconciled into inconsistent output.
- Define and document a conflict-resolution policy: last-writer-wins is the default
  anti-pattern; prefer explicit conflict objects requiring resolution.

---

## What This Is *Not*

- Not a sandbox. (A lightweight process sandbox — seccomp + namespaces — remains a
  reasonable *last-mile* defense for locally-generated code, but it is out of scope
  for v1 and explicitly not the primary mechanism.)
- Not a VM/desktop computer-use framework.
- Not a replacement for the agent orchestration framework — it is the substrate the
  orchestrator sits on top of.

### What Phase 0 hardened into code

The "spec" work for Phase 0 was resolved implicitly during Phase 1 implementation:
- **Capability token schema** is formal and typed: `Capability` dataclass with
  `DataSelector`, `operations: frozenset[str]`, `agent_id`, `issued_by`, `not_after`,
  HMAC signing, and attenuation invariants enforced at the type level.
- **Audit log entry schema** is concrete: `AuditEntry` with `seq`, `ts`, `actor`,
  `action`, `target`, `capability_id`, `details`, `prev_hash`, `entry_hash`.
- **Tamper-evidence scheme** is implemented: SHA-256 hash chain from a genesis hash;
  `AuditLog.verify()` recomputes the chain and raises `TamperError` on any break.
- **Threat model document** is not yet written as a standalone artifact.

### Phase 1 — Immutable Store + Audit Log (the foundation)
- [x] Append-only versioned store with time-travel reads. *(SQLite-backed `ImmutableStore`)*
- [x] Hash-chained append-only audit log with a write-only agent interface. *(SHA-256 chain, `AuditLog`)*
- [x] Reference SDK: `read()`, `append()`, `tombstone()`, `history()`, `scan()`, `verify()`.

### Phase 2 — Capability Broker + Enforcement
- [x] Concrete `CapabilityBroker`: issuance, revocation, TTL — every decision audited. *(SQLiteCapabilityBroker)*
- [x] Data access layer that *enforces* tokens (deny-by-default). *(EnforcedStore)*
- [ ] Agent identity via SPIFFE/SPIRE or signed agent certs. *(v1 uses agent_id in token; production upgrade documented in THREAT_MODEL.md)*

### Phase 3 — LLM Capability Generation + Verification
- [x] `LLMCapabilityGenerator` backed by `OllamaLLMClient` (local model, zero new deps).
- [x] `LLMVerifierAgent` running in a *separate* model context; enforces separate-instance check at construction. Reject/narrow loop.
- [x] End-to-end: NL policy → generated capability → verified → issued → enforced. (`CapabilityPipeline`)

### Phase 4 — Memory Layer + Discrepancy Detection
- [x] `SQLiteMemoryLayer`: provenance-tagged writes `(agent_id, source_version, ts)`.
- [x] Snapshot-based: `snapshot(agent_id)` returns consistent point-in-time view.
- [x] Divergence detector: unverified-source and cross-agent-conflict detection against audit log.

### Phase 5 — Reference Multi-Agent Demo
- [ ] A 3-agent scenario showing: a malicious/buggy agent is contained by the data
      layer (not a sandbox), ground truth is recoverable from the audit log, and a
      cross-agent discrepancy is detected and surfaced.

## Open Questions

- Performance overhead of capability checks on the hot path — caching strategy for
  capability decisions without weakening revocation.
- How fine-grained should data selectors be (row / column / cell / semantic)?
- Verifier agent reliability — how to bound the false-negative rate (approving an
  over-permissioned capability) and whether a formal/static check should backstop it.
- Standardization — does this map onto emerging standards (MCP for tool interfaces,
  agentskills.io) or need its own?
- Conflict resolution semantics for the consistency model — what is the default and
  what is configurable?

## Tech Stack (proposed, v1)

  audit log hot path.
